# FamilyOS 3.4.0 Full-App Post-Release Audit

Date: 2026-07-16
Audited release: 3.4.0 at `a4482b4f1f7303a2a2bb1c395a70748ec6529f6a` (`v3.4.0`)
Decision: **Ready with conditions**
Overall health: **7.4/10**

## Executive summary

Release 3.4.0 is usable for continued household use. The production deployment is Ready, the production read-only verifier passed, the optimized build and bundle marker scan passed, 165 unit tests passed, 69 of 81 full Playwright regression tests passed, all required smoke projects passed, and representative automated accessibility checks passed.

The next feature release should not begin until the P1 release-gate defect is corrected and the P2 regression and authorization gaps are closed. `pnpm run check` currently fails because `scripts/validate-supabase-bootstrap.mjs` still asserts Release 3.3.0 and is not aligned with the 3.4.0 manifest. The full browser suite also has 12 repeated failures caused by three Home assertions that no longer match Release 3.4 behavior. No P0 security, data-loss, production-targeting, or secret-exposure defect was found, but multi-household and role-level authorization were not dynamically proven during this audit.

## Database and security audit

Passed evidence:

- Test target controls require explicit confirmation, expected project matching, remote-test opt-in, and a blocked production ref.
- 44 safety assertions passed.
- Release 3.4 static migration assertions cover four new tables, lifecycle states, RLS enablement, and household role checks.
- Manifest includes Release 3.4 as the final approved migration and blocks production.
- Bundle scan found no development-only secret markers.
- Private Storage authenticated upload, metadata RLS, and signed URL validation passed.
- Production validation was read-only.

Limitations:

- A dynamic two-household attack matrix was not available and was not created. Cross-household SELECT/INSERT/UPDATE/DELETE, guessed IDs, payload tampering, and direct REST calls therefore remain unproven in this audit.
- Owner/adult/member/child behavior was not dynamically tested with separate identities.
- Hosted blank-database initialization could not be certified because the dedicated project already contained the approved demo household. The initializer correctly refused to treat it as empty.
- The Release 3.4 validator is primarily source-pattern based and does not replace catalog- and identity-level RLS tests.

No evidence of a service-role key in the browser bundle, anonymous access, cross-household exposure, or production mutation was observed. Absence of evidence is not a dynamic authorization pass.

## Recommendation-engine assessment

The deterministic engine has strong unit coverage for confidence scoring, confidence/priority separation, workload reduction, adaptive cooldown, material-change bypass, grouping, dependency resolution, duplicate prevention, and effectiveness metrics. The migration is additive and defines household-local feedback, effectiveness, learning, and dependency records.

The material gap is end-to-end persistence and authorization. There is no Release 3.4 Playwright suite proving lifecycle transitions, feedback, settings, reset/export scoping, role restrictions, dependency ordering, or no-silent-mutation behavior. Recommendation-engine status is therefore **technically promising but conditionally verified**.

- Production weather, desktop/mobile load, console, and bundle checks passed.
- Main bundle is 230.35 kB gzip; this audit found no release-specific budget failure.
- Weather sets `s-maxage=1800, stale-while-revalidate=3600` and uses a 6.5-second abort timeout.

- The known production `[DEP0169] url.parse()` warning could not be attributed from available logs. No `url.parse()` call exists in `api/weather.js` or `api/calendar.js`, so dependency/platform code is the likely source, but this is an inference. The locally reproduced warning is separate: `[DEP0176] fs.F_OK` from `react-dev-utils@12.0.1`. The known warning is non-blocking absent functional errors, but its stack trace should be captured before the next release.
- Runtime error clusters, 4xx/5xx trends, slow routes, duplicate requests, and request-volume analysis could not be quantified from the available read-only verifier. This is an observability gap.

## Defect register

### AUD-340-001: Release aggregate check cannot pass

- Severity: P1 High
- Area/environment: release validation, local/CI
- Preconditions: Release 3.4.0 checkout
- Reproduction: run `pnpm run check` or `pnpm run test:db-bootstrap`.
- Expected: reconciliation validates the 3.4.0 manifest and completes.
- Actual: assertion reports `'3.4.0' !== '3.3.0'` at `scripts/validate-supabase-bootstrap.mjs:10`.
- Evidence/root cause: script retains 3.3 version/count assertions and final copy.
- Resolution: update reconciliation assertions/counts/copy for 3.4 and add `test:db-3.4` explicitly to `check`; confirm nonzero child exits cannot be masked by audit wrappers.
- Next-release blocker: Yes.
- Regression test: Existing script should become the gate.

### AUD-340-002: Full Playwright regression suite has 12 stale Home assertion failures

- Severity: P2 Medium
- Area/environment/viewport: E2E tests; test Supabase; desktop, 390 px mobile, tablet, dark.
- Reproduction: `pnpm run test:regression`.
- Expected: 81/81 pass.
- Actual: 69 pass; Release 2.3 ambiguous Family Brief locator, Release 3.1 missing former Relationship Focus Home copy, and Release 3.2 missing former Executive summary Home copy fail in all projects.
- Evidence/root cause: Release 3.4 Home layout/copy changed while older tests retained presentation-specific preconditions. Captured screenshot shows the 3.4 Family Brief is rendered.
- Resolution: use stable roles/test IDs and navigate directly to owning modules before asserting module behavior; separately decide whether Relationship Focus/Executive summary removal was intended.
- Next-release blocker: Conditional. Must be triaged and green before feature work merges.
- Regression test: Update the existing tests.

### AUD-340-003: Release 3.4 critical lifecycle and authorization flows lack dynamic coverage

- Severity: P2 Medium
- Area: recommendations/database security
- Environment: hosted test
- Expected: multi-household and role matrix proves every lifecycle mutation, reset, export, metrics, and feedback boundary.
- Actual: source-pattern and unit assertions exist; no dedicated 3.4 browser/API/RLS suite exists.
- Root cause: Release 3.4 validation added static migration/engine checks without an authenticated multi-identity harness.
- Resolution: add a compact two-household owner/adult/member/child API suite plus one browser lifecycle happy path.
- Next-release blocker: Yes for security-critical recommendation changes; otherwise a condition before Release 3.5.
- Regression test: Yes.

### AUD-340-004: Blank hosted database and 3.3-to-3.4 upgrade path not dynamically certified in this audit

- Severity: P2 Medium
- Area: migrations/test operations
- Environment: hosted test/local
- Actual: hosted verifier found the demo household; local Pool validator was blocked by unavailable Docker. Static 3.4 assertions and safety tests passed.
- Resolution: provide a documented disposable hosted reset workflow or a second disposable database, then run blank chain, 3.3 upgrade, repeat validation, and pre/post row-count checks.
- Next-release blocker: Condition for the next schema-changing release.
- Regression test: Yes.

### AUD-340-005: Known DEP0169 runtime warning lacks source stack and trend evidence

- Severity: P3 Low
- Area: production observability
- Actual: warning was previously observed on weather/calendar; current read-only verifier passed and first-party code contains no `url.parse()`.
- Resolution: capture one runtime stack with Vercel function/runtime version and identify the transitive package; apply only a narrow safe upgrade or platform configuration change.
- Next-release blocker: No, unless accompanied by request failures.
- Regression test: No; operational log check.

### AUD-340-006: Aggregate `check` omits the Release 3.4 validator

- Severity: P2 Medium
- Area: release automation
- Actual: `test:db-3.4` exists and passes but is not invoked by `check`.
- Resolution: add it immediately after the 3.3 validator and keep future release gates version-aligned.
- Next-release blocker: Yes as part of AUD-340-001.
- Regression test: Script-level gate.

## Proposed 3.4.1 stabilization scope

1. Fix the 3.4 reconciliation gate and include `test:db-3.4` in `check`.
2. Update the three stale Playwright workflows and require 81/81 green.
3. Add dynamic Release 3.4 recommendation lifecycle and two-household/role authorization tests.
4. Run a clean blank/3.3-upgrade database validation in a disposable environment.
5. Capture and attribute the production DEP0169 stack; defer broad dependency upgrades.

Safe to defer: exhaustive cosmetic viewport sweeps, broad CRA dependency modernization, comprehensive performance telemetry, and low-risk copy refinements, provided the conditions above are met.

## Release recommendation

FamilyOS 3.4.0 is **ready for continued household use with conditions**. Stabilize the release controls and security-critical test coverage before beginning or merging the next feature release. There is no evidence supporting an emergency rollback.
