# Local Seed Startup Fix

## Error Encountered

Local Supabase startup applied both migrations successfully, then failed while seeding data from:

- `supabase/seed.sql`

The reported error was:

```text
Replace seed_user_id with your Supabase auth user UUID before running seed.sql.
```

## Root Cause

`supabase/config.toml` generated by `supabase init` enables automatic seeding with:

```toml
[db.seed]
enabled = true
sql_paths = ["./seed.sql"]
```

The existing seed file is optional starter data and intentionally used a placeholder user id:

```sql
00000000-0000-0000-0000-000000000000
```

That guard is useful for manual seeding, but it blocked local Supabase startup because `supabase start` runs the seed file automatically after migrations.

## Files Changed

- `supabase/seed.sql`

## Exact Fix Made

The placeholder guard now skips optional starter data with a notice instead of raising an exception:

```text
Skipping optional starter data. Replace seed_user_id with your Supabase auth user UUID before running seed.sql manually.
```

No starter data is inserted while the placeholder UUID remains in place.

## Production Safety

No production commands were run.

No remote Supabase project was linked.

No application code was changed.

## Next Local Retry Commands

Run these only against the local Supabase environment:

```powershell
pnpm dlx supabase stop --no-backup
pnpm dlx supabase start
```

## Commands Not To Run

Do not run:

```powershell
supabase link
supabase db pull
supabase db push
```

## Follow-Up

After local Supabase starts, create a local test user through the app or Supabase Studio before manually loading optional starter data. If starter data is needed, replace `seed_user_id` with that local auth user's UUID first.
