# Calendar And Branding Polish Summary

Date: June 29, 2026
Branch: `feature/household-foundation`

## Executive Summary

This sprint tightened the Google Calendar connection path and verified the current Family OS app branding assets. The Calendar implementation already uses the correct Google Identity Services popup token flow, but local connection can fail if Google Cloud does not include the exact browser origin, especially `http://127.0.0.1:3000`.

No production migrations were applied, no remote Supabase commands were run, no Google Cloud settings were changed automatically, and no secrets were added to the repo.

## Calendar Issue Diagnosis

Family OS uses:

- implementation: `src/hooks/useGoogleCalendar.js`
- OAuth API: Google Identity Services `window.google.accounts.oauth2.initTokenClient`
- scope: `https://www.googleapis.com/auth/calendar.readonly`
- client id source: `REACT_APP_GOOGLE_CLIENT_ID`
- calendar id source: `REACT_APP_GOOGLE_CALENDAR_ID`
- local app origin during validation: `http://127.0.0.1:3000`
- redirect URI: none generated by the app for this popup token flow

The likely local connection issue is Google Cloud OAuth origin configuration, not a redirect callback bug. Because the app uses the browser token popup flow, Google validates the exact JavaScript origin serving the app. If the app is opened at `http://127.0.0.1:3000`, then `http://127.0.0.1:3000` must be listed in Authorized JavaScript origins. Having only `http://localhost:3000` is not enough.

Other possible causes now diagnosed or handled:

- missing/placeholder `REACT_APP_GOOGLE_CLIENT_ID`
- missing/placeholder `REACT_APP_GOOGLE_CALENDAR_ID`
- Google Identity Services script not loading
- popup blocked
- user denies access
- token response missing an access token
- Calendar API errors after OAuth succeeds

## Google Cloud Manual Checklist

Do this manually in Google Cloud Console. Do not store any Google secret in the repo.

### OAuth Client

- Open **APIs & Services > Credentials**.
- Select the OAuth 2.0 Client ID used by `REACT_APP_GOOGLE_CLIENT_ID`.
- Confirm the client type is **Web application**.

### Authorized JavaScript Origins

Add every exact app origin used to open Family OS:

```text
http://localhost:3000
http://127.0.0.1:3000
https://YOUR_VERCEL_PRODUCTION_DOMAIN.vercel.app
https://YOUR_CUSTOM_DOMAIN
```

Notes:

- No trailing slash.
- `localhost` and `127.0.0.1` are different origins.
- Vercel preview URLs must be added individually if Calendar sync is needed there.
- Wildcards are not supported for Google OAuth JavaScript origins.

### Authorized Redirect URIs

The current Family OS Calendar flow does not require redirect URIs because it uses Google Identity Services popup token auth.

If Google Cloud requires values for this Web client, add only exact app URLs that are intentionally supported:

```text
http://localhost:3000
http://127.0.0.1:3000
https://YOUR_VERCEL_PRODUCTION_DOMAIN.vercel.app
https://YOUR_CUSTOM_DOMAIN
```

### OAuth Consent Screen

- Publishing status can remain Testing for local/private use.
- Add the Google accounts that will test Calendar sync as test users if the app is in Testing mode.
- App name should match FamilyOS or Family OS.
- Required scope:

```text
https://www.googleapis.com/auth/calendar.readonly
```

### APIs

- Ensure Google Calendar API is enabled for the same Google Cloud project as the OAuth client.

### Local Environment

`.env.local` should contain:

```env
REACT_APP_GOOGLE_CLIENT_ID=<Google OAuth Web client id>
REACT_APP_GOOGLE_CALENDAR_ID=primary
```

Use a specific calendar id instead of `primary` only if the target is a shared calendar.
