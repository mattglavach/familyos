# Platform AI Context

## Release 2.0 implementation

Release 2.0 implements the provider-neutral Context Engine and Prompt Builder described in `docs/architecture/CONTEXT_ENGINE.md`. The earlier Household Context Service remains the dashboard compatibility contract. New AI-facing work must use module contributors and `familyos.context-engine` version 2.0.

The AI Workspace is the only active prompt-generation surface. It previews and copies prompts without automatic transmission, stores only trace metadata on the device, and never writes AI output to FamilyOS records.

## Release 1.8 implementation boundary (historical)

The broader Context Engine remains an architectural pattern. The Household Context Service is the first reusable implementation and emits `familyos.household-context` version 1.0. It is deterministic and does not call ChatGPT. Future ChatGPT reasoning may consume the structured output, but recommendations remain proposals requiring human review and acceptance.

> Canonical AI and data ownership are defined in `docs/governance/FamilyOS_Project_Instructions.md`. This document provides specialized AI-context architecture detail.

## Purpose
Define how Family OS prepares structured context for AI assistance.

## Goals
- Keep AI grounded in household records.
- Make module summaries consistent.
- Separate facts from recommendations.
- Prevent AI from inferring from raw data alone.

## Scope
This document describes the AI context layer. It does not define prompts for a specific model or add implementation code.

## Architecture
AI should consume structured context generated by each module and the platform. Raw tables alone are not enough. Context builders should summarize relevant records into stable, reviewable payloads before prompt assembly.

## Context Builders
Each module should provide a context builder that returns structured facts, open questions, risks, and suggested next actions.

Examples:

- Pool: latest chemistry metrics, equipment assets, maintenance tasks, chemical additions, and upcoming reminders.
- Finance: goals, balances, account metrics, review tasks, policy documents, and upcoming financial reminders.
- College: student profile, application goals, test scores, deadlines, study tasks, visits, and documents.
- Home: properties, home assets, maintenance tasks, warranties, vendors, documents, and open reminders.
- Garden: garden assets, plants, seasonal tasks, metrics, notes, and upcoming maintenance.
- Medical: people, appointments, documents, reminders, notes, and time-sensitive follow-ups.

## Prompt Assembly
Prompt assembly should combine:

- household context
- relevant module contexts
- active tasks and reminders
- upcoming events
- recent metrics
- documents and notes summaries
- user request
- safety and scope instructions

The final prompt should distinguish known facts from recommendations.

## AI Suggestions
AI-generated actions should be saved as drafts or records with source `ai_suggested`. The user should confirm before suggestions become active tasks, reminders, or recurring workflows.

## Future Memory Integration
Future memory should store approved durable facts, preferences, and decisions. Memory should not override the database and should not contain unreviewed AI assumptions.

## Design Decisions
- The database remains the source of truth.
- AI consumes structured context, not raw data alone.
- Context builders are module-owned but platform-shaped.
- AI recommendations require review before changing operational records.

## Future Considerations
- Context freshness tracking.
- User-approved memory.
- Citation of source records in AI responses.
- Cross-module reasoning for weekly and monthly reviews.
